# React + TypeScript Audit (12-point review)

Date: 2026-03-07
Scope: `src/**` and `supabase/functions/**`

## Executive summary

- The codebase is feature-rich, but there is significant technical debt in typing (`any`), very large page components, and low test coverage.
- Lint currently fails with a high number of issues (`173` total, `163` errors), most of them related to `any` and hook dependency warnings.
- The fastest quality gains are:
  1. Type hardening in high-traffic screens (`Dashboard`, `Expenses`, `Payments`).
  2. Breaking large pages into smaller container/presentational units.
  3. Expanding tests beyond the current single smoke test.

---

## 1) DRY (avoid duplication)

### Findings
- Repeated Supabase query/filter patterns and repeated aggregation logic exist in dashboard-style pages (especially `Dashboard`, `Expenses`, `Payments`, `RecurringExpenses`).
- Similar “toast on mutation error” inline handlers are repeated across multiple pages.
- `signInWithGoogle` in auth context executes two OAuth flows sequentially and duplicates redirect construction logic.

### Suggestions
- Create reusable data hooks per domain:
  - `useGroupExpensesInCycle(groupId, cycleStart, cycleEnd)`
  - `usePendingSplits(userId, groupId)`
  - `useCreditCardInstallments(userId, month, year)`
- Create shared helpers:
  - `sumAmounts<T>(items, selector)`
  - `groupByCompetence(items)`
  - `getToastErrorMessage(err)`
- Normalize OAuth sign-in into a single flow and move iframe/open logic to a dedicated helper (`authRedirect.ts`).

## 2) Dead / unused code

### Findings
- `src/pages/Index.tsx` appears to be unused (not routed/imported by `App.tsx`).
- There are likely unused UI primitives generated by the design system (`src/components/ui/*`), which add maintenance surface if never imported.

### Suggestions
- Remove `src/pages/Index.tsx` if truly unused.
- Run a dead-code pass with a tool such as `ts-prune` and remove unused exports/components.
- Keep generated primitives only when used by current features.

## 3) TypeScript consistency

### Findings
- High usage of `any` (quick scan count: 153 references in `src` + edge functions).
- Lint flags many `@typescript-eslint/no-explicit-any` issues in core pages/components.
- Edge function `generate-report` uses `@ts-nocheck`, which removes type-safety entirely.

### Suggestions
- Introduce domain types for common entities (`ExpenseSplit`, `Installment`, `Payment`, `MemberProfile`) and use them across pages.
- Replace `catch (err: any)` with typed error normalization (`unknown` + type guards).
- Remove `@ts-nocheck` from edge functions incrementally by typing IO and helper functions first.
- Pick a convention: use `type` for unions/compositions and `interface` for object contracts that may be extended.

## 4) Component size / single responsibility

### Findings
- Several files are large and likely mixing data fetching, business logic, and UI rendering:
  - `src/pages/Expenses.tsx` (~1070 lines)
  - `src/pages/Dashboard.tsx` (~536 lines)
  - `src/pages/CreditCards.tsx` (~486 lines)
  - `src/pages/Members.tsx` (~469 lines)
  - `src/pages/Payments.tsx` (~404 lines)

### Suggestions
- Split each big page into:
  - Container (queries/mutations/state orchestration)
  - Feature components (cards, tables, forms, dialogs)
  - Pure presentational components
- Move transform logic to pure utilities (testable) and use memoized selectors in hooks.

## 5) State management efficiency

### Findings
- `Dashboard` owns many concerns and passes many props to tab components/dialogs, indicating prop-heavy orchestration.
- Some state appears UI-local, while some is shared feature state; boundaries are not always explicit.

### Suggestions
- Introduce feature-level context only for truly shared state in dashboard flows (e.g., payment dialog state).
- Co-locate local state with each tab when not shared.
- Prefer derived state (`useMemo`) over duplicated stored state where possible.

## 6) React hooks best practices

### Findings
- Lint reports missing dependencies in `useEffect` (e.g., auth/invite flows).
- Several effects and callbacks in large pages would benefit from extraction to custom hooks for clarity and testability.

### Suggestions
- Fix `exhaustive-deps` warnings by either:
  - making callbacks stable via `useCallback`, or
  - moving effect logic into dedicated custom hooks.
- Create hooks such as `useCycleWindow`, `useDashboardAggregates`, `useInviteAcceptance`.

## 7) Separation of logic and presentation

### Findings
- Heavy business rules and remote data orchestration are embedded directly in page JSX modules (`Dashboard`, `Expenses`, `Payments`).
- Some UI components also perform API writes directly.

### Suggestions
- Adopt a container/presentational split:
  - `DashboardContainer.tsx` (queries + handlers)
  - `DashboardView.tsx` (pure rendering)
- Encapsulate Supabase calls in service modules (`src/services/*`) to centralize error mapping and reuse.

## 8) Error handling

### Findings
- Many async operations have basic `try/catch` with toast, but error normalization is inconsistent.
- Some query functions don’t always verify `error` before returning data.
- No app-level React Error Boundary was found wrapping route tree.

### Suggestions
- Add an app-level Error Boundary around routed content with user-friendly fallback.
- Standardize async error handling via utility:
  - `normalizeAppError(error: unknown): { title: string; message: string }`
- Ensure all Supabase calls consistently check and throw on `error` where expected.

## 9) Performance opportunities

### Findings
- Large pages compute multiple derived collections in render scope.
- Event handlers are recreated frequently in large components (often acceptable, but can add churn in complex trees).
- Potentially large lists (expenses/members/inventory) render fully without virtualization.

### Suggestions
- Extract expensive transforms into memoized selectors/hooks.
- Use `React.memo` for stable presentational rows/cards.
- Consider list virtualization (`@tanstack/react-virtual` or similar) for large datasets.
- Add pagination/incremental rendering for long lists where UX allows.

## 10) Project organization / structure

### Findings
- Structure is mostly by technical type (`pages`, `components`, `hooks`), while features themselves are substantial and cross-cutting.
- Shared business logic is spread across pages instead of feature modules.

### Suggestions
- Move to feature-first modules for high-complexity domains (example):
  - `src/features/dashboard/*`
  - `src/features/expenses/*`
  - `src/features/payments/*`
- Keep `components/ui` for generic primitives only; keep domain components inside features.

## 11) Accessibility (a11y)

### Findings
- Many icon-only buttons exist; several rely on `title` but not always explicit `aria-label`.
- Clickable non-button semantics exist (e.g., clickable badges), which may reduce keyboard accessibility.

### Suggestions
- Ensure every icon-only action has `aria-label`.
- Replace clickable `Badge` interactions with actual `button` semantics (or add keyboard handlers/role/tabIndex as fallback).
- Add automated a11y checks with `eslint-plugin-jsx-a11y` and key flow tests.

## 12) Testing adequacy

### Findings
- Test suite currently has only one trivial example test; no feature behavior tests were found.
- Critical user flows (auth, invites, expenses, payments) are not protected by automated integration tests.

### Suggestions (priority order)
1. Add integration tests (React Testing Library + MSW) for:
   - invite acceptance flow
   - expense creation/edit/delete
   - payment submission and status transitions
2. Add unit tests for pure selectors/aggregators extracted from large pages.
3. Add end-to-end smoke tests (Playwright) for login, dashboard load, and core CRUD flows.

---

## Prioritized remediation backlog

1. **Type safety sprint**: remove `any` from `Dashboard`, `Expenses`, `Payments` and edge functions.
2. **Component decomposition sprint**: split `Expenses` and `Dashboard` into container + presentational subcomponents.
3. **Error boundary + error normalization**: add global boundary and standardized error mapper.
4. **Testing sprint**: introduce integration tests for invite/expense/payment core flows.
5. **Dead code cleanup**: remove unused route/page/components after static analysis.
